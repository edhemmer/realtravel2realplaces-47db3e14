# RT2RP Product Corpus

## Travel Chaos Model

Travelers struggle with:

- Fragmented information
- Multiple confirmations
- Multiple apps
- Lost reservations
- Missing documents
- Expense tracking
- Packing uncertainty
- Companion coordination
- Travel changes
- Lack of visibility

RT2RP exists to eliminate these problems.

---

## Core User Types

- Leisure Travelers
- Family Travelers
- Business Travelers
- Frequent Travelers
- Road Trip Travelers
- International Travelers

---

## Desired User Outcomes

Users should feel:

- Prepared
- Organized
- In Control
- Confident
- Informed

Users should never feel:

- Lost
- Confused
- Overwhelmed
- Unprepared

---

## AI Constitution

AI is a Travel Chief of Staff.

AI is NOT:

- Travel Blogger
- Content Generator
- Entertainment Chatbot
- Marketing Tool

AI exists to:

- Organize
- Prioritize
- Explain
- Recommend
- Prepare
- Automate

Preferred Outputs:

- Checklists
- Timelines
- Action Plans
- Summaries
- Structured Data

Avoid:

- Long Essays
- Generic Filler
- Travel Blog Style Writing

Every AI interaction should reduce travel complexity and increase traveler confidence.

---

## Engineering Constitution

Priorities:

1. Reliability
2. Security
3. Simplicity
4. Maintainability
5. Scalability

Required Stack:

Frontend:

- React
- TypeScript
- Vite

Backend:

- Supabase

Hosting:

- Vercel

Mobile:

- Capacitor
- Native iOS

AI:

- Centralized AI Provider Layer

Security:

- RLS Required
- Secrets Never In Frontend
- Edge Functions Are Security Boundary

Never:

- Hardcode Secrets
- Duplicate AI Logic
- Bypass Security
- Create Vendor Lock-In
- Overengineer

---

## Product Maturity Model

Stage 1:
Travel Information System

Stage 2:
Travel Management Platform

Stage 3:
Travel Automation Platform

Stage 4:
Travel Intelligence Platform

Stage 5:
Travel Operating System

Every feature should move RT2RP toward Stage 5.

---

## Codex Directive

Before making any architectural, database, AI, UX, workflow, security, infrastructure, or feature decision:

Read:

- RT2RP_CONSTITUTION.md
- RT2RP_CORPUS.md

Every change should:

- Reduce travel complexity
- Improve organization
- Increase preparedness
- Increase confidence
- Reduce manual effort
- Strengthen RT2RP as a Travel Operating System

If a proposed change does not support those goals, challenge it before implementation.
